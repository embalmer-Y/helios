整体分层架构图

当前结构遵循“编排在主循环、能力在子系统、接触世界的实现统一收口到 helios_io”这一原则。图中强调拥有实现的边界，而不是历史兼容导出位置。

Layer Ownership Map

The current structure follows one rule above all: orchestration lives in the main loop, subsystem capability lives in dedicated packages, and all world-facing implementation is owned by helios_io.

外部接口层QQ、STT、TTS、Vision、Dashboard 等输入输出表面
I/O 边界层通道抽象、协议适配、SEC 评估、被动回复、行为执行
编排层Helios 主循环、TickGuard、HeliosState、EventSource
内部能力层情感底盘、记忆、认知、调节
External SurfaceQQ, STT, TTS, Vision, Dashboard, and other external touchpoints
I/O BoundaryChannel abstraction, protocol adapters, SEC evaluation, passive reply, behavior execution
OrchestrationMain loop, TickGuard, HeliosState, EventSource, trigger merge
Internal CapabilityAffective substrate, memory, cognition, regulation
主循环是编排层，不是所有能力的实现归属层。它持有对象引用，但不吞并各子系统所有权。
`helios_io` 是唯一外部边界层。新增协议、通道、内容外发组件都应优先进入该目录。
`core` 只保留运行基础设施。兼容导出可以存在，但不改变实际拥有实现的位置。
The main loop is an orchestration layer, not the owner of every subsystem implementation.
`helios_io` is the one external boundary layer. New protocols, channels, and outward generation should enter there first.
`core` remains transport-agnostic runtime infrastructure. Compatibility exports do not change real ownership.

Tick 生命周期流程图

下面的主流程对应 `Helios._tick_once()` 的当前真实顺序。它是新版系统行为的事实定义，也是文档和测试对齐的基准。

Tick Lifecycle Flow

This flow mirrors the current order inside `Helios._tick_once()`. It is the most concrete execution contract behind the active architecture docs.

A
统一状态起点
所有共享运行态字段都先进入 `HeliosState`，避免跨模块读私有对象造成隐式耦合。
B
闭环中段
Affect、ICRI、drives、thinking 和 memory 在同一 tick 内互相影响，构成内部闭环。
C
边界后段
回复生成与行为执行都在内部状态已经稳定后才触发，确保输出是内部过程的结果，而不是入口条件的直接镜像。
A
Shared state first
Tick-scoped runtime facts enter `HeliosState` before subsystem work begins.
B
Closed internal loop
Affect, ICRI, drives, thinking, and memory influence one another within the same tick.
C
Boundary work later
Reply generation and behavior execution run after internal state has stabilized for the tick.

关键对象流

新版结构的重点不只是模块名，还在于对象如何穿过模块。对象流决定了哪个层在产生事实，哪个层只是在消费事实。

Key Object Flow

The important part of the new structure is not only where modules live, but which objects move across them and who owns each fact.

HeliosState每个 tick 的单一事实源，聚合 affect、ICRI、drives、thinking、behavior、I/O availability 和稳定性指标。
ChannelMessage由 `helios_io` 统一承载跨边界消息，主循环和调节层无需关心协议细节。
CognitiveImpactProfile把多模态输入的影响投影到统一语义面，供 Phi 与上层认知复用。
Behavior Command由 regulation 产出动作意图，再由 LimbDecisionBridge 映射为执行优先级与命令生命周期。
HeliosStateThe single source of truth for each tick, aggregating affect, ICRI, drives, thinking, behavior, availability, and stability metrics.
ChannelMessageThe common external-boundary message type owned by `helios_io`, hiding protocol details from the main loop.
CognitiveImpactProfileA normalized structure for sensory, cognitive, self-related, and novelty impact that can feed phi and higher reasoning.
Behavior CommandAn execution-layer command derived from regulation output through priority mapping and queue lifecycle.

边界规则

维护新版结构时，先判断职责归属，再判断实现位置。

Boundary Rules

When extending the system, decide ownership first and implementation placement second.

属于 helios_io 的内容协议、通道、路由、编码、外发文本生成、外部行为落地。
属于 core 的内容EventSource、状态快照、触发合并、tick 保护等传输无关基础设施。
属于内部能力层的内容记忆、认知、调节与情感模型，不直接拥有协议实现。
属于 research 文档的内容解释实现边界与运行逻辑，而不是充当旧方案堆场。
Belongs in helios_ioProtocols, channels, routing, serialization, outward text generation, and external action execution.
Belongs in coreEvent sources, state snapshots, trigger merging, and tick protection that remain transport-agnostic.
Belongs in internal capability layersMemory, cognition, regulation, and affective models that do not directly own protocol code.
Belongs in research docsExplanation of boundaries, runtime behavior, and evolution rules rather than a dump of obsolete ideas.
