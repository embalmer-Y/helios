# Helios Soul-Core Enhance 需求文档

> Status: Finalized
> Audience: 架构维护者、功能开发者、研究开发者
> Scope: 本轮 soul-core enhance 的正式需求、边界、约束与验收标准

## 1. 文档目标

本文件用于把当前确认的 9 项增强需求整理为可执行的正式需求集合，作为后续低层设计、任务拆分、研究收集和代码实现的统一基线。

本文件关注以下问题：

- 这轮增强到底要解决什么问题
- 哪些能力属于必须交付，哪些属于后续扩展
- 哪些模块可以直接实现，哪些模块必须先进行学术研究
- 哪些运行时决策可以由 LLM 提议，哪些必须由系统策略层与执行层控制

## 1.1 原始需求编号映射说明

为避免后续讨论把“原始需求编号”和“实施阶段/任务编号”混为一谈，本轮文档固定采用如下映射：

1. 原始需求第 5 条对应本文 `R5`：前意识/非意识行为路径。
2. 原始需求第 6 条对应本文 `R6`：人格影响模块。
3. 原始需求第 7 条对应本文 `R7`：行为模块数据库化与动态扩展。

任务书中的 `T6/T7/T8` 是实施阶段编号，不直接等于原始需求 6/7 的编号。

## 2. 背景

Helios 当前已经具备持续运行闭环、情感更新、驱动估计、被动回复、主动行为调节、基础多通道接入与多层记忆能力，但仍存在以下结构性限制：

1. 被动回复路径仍以单条 SEC 阈值判断为主，尚不能表达“一个具备人格、情绪、注意力与需求的主体是否想回应”。
2. 主动行为选择仍以固定动作池和较少的加权因素为主，尚未把人格、神经化学、时间动力学和上下文约束系统性纳入。
3. Tick 目前主要承担运行节拍职责，尚未被完整建模为主体的主观时间流与慢变量驱动机制。
4. 主动输出虽然经过行为执行抽象，但最终外发仍以 QQ 文本为主，未形成 channel-agnostic 的统一行为表达能力。
5. 输入到达意识层之前，尚不存在单独的前意识/非意识行为通路。
6. 人格虽然已经存在状态表示，但尚未以独立影响模块的形式作用于交互和行为选择。
7. 行为系统仍以代码内置动作池为主，不支持数据库化管理、来源记录、动态扩展与运行时学习闭环。
8. IO channel 目前只有最小输入/输出接口，尚不具备自描述能力、能力查询能力和标准化 ops 描述。
9. 长期记忆仍以文件持久化为主，尚未建立面向复杂检索、追踪、压缩审计和 LLM 记忆利用的数据库后端。

## 3. 本轮目标

本轮增强的总目标是：

把 Helios 从“具备部分情感与行为机制的持续运行代理”推进为“具备统一动作提议、统一策略评估、统一执行总线、时间动力学、人格投影、可扩展行为注册表、可描述通道与数据库化长期记忆基础设施的 soul-core”。

## 4. 核心原则

### 4.1 主体性原则

被动回复和主动行为都应体现主体内部状态，而不应只是单纯地对输入做模板化响应。

### 4.2 统一动作原则

系统中的被动回复、主动行为、前意识行为都必须收敛到统一的动作提议与执行模型，而不是分别维护多套决策路径。

### 4.3 LLM 提议而非裸执行原则

LLM 可以产生结构化行为提议，但不能直接绕过策略层、权限层、能力层和执行层去控制外部通道、硬件或程序。

### 4.4 通道无关原则

行为定义不应写死到单一通道，行为应描述意图、模态、执行约束和首选能力，再由系统根据可用 channel 做绑定和路由。

### 4.5 研究先行原则

涉及“前意识/非意识行为”与“人格影响模块”的实现，必须先完成权威学术来源收集、摘要整理和架构映射，再开始编码。

### 4.6 闭环反馈原则

行为选择、执行结果、用户反馈、通道回执、记忆写入、内部状态变化都必须进入统一的反馈环，用于后续分析和学习。

## 5. 范围与非目标

### 5.1 本轮范围

本轮范围包含：

1. 新的交互表达评判逻辑
2. 主动触发与行为选择增强
3. Tick/时间动力学增强
4. 主动输出从单一 QQ 路由升级为 channel-agnostic 行为执行
5. 前意识/非意识行为研究与模块设计
6. 人格影响模块研究与模块设计
7. 行为系统数据库化与动态扩展设计
8. IO channel 描述与 ops 能力增强
9. 记忆系统数据库化架构设计

### 5.2 本轮非目标

本轮不要求：

1. 一次性实现所有新行为和所有新通道。
2. 一次性替换掉现有全部记忆文件格式。
3. 让 LLM 在无约束情况下自动注册并激活任意行为。
4. 在没有研究输入的情况下直接落地前意识/非意识行为模型。

## 6. 术语

### 6.1 ActionProposal

由策略模块或 LLM 生成的结构化动作提议，描述候选行为、理由、约束、建议模态、候选 channel 和预期目标。

### 6.2 BehaviorSpec

行为注册表中的正式行为定义，包含行为名称、用途、参数 schema、可用 channel、执行代价、前置条件、冷却、来源与版本信息。

### 6.3 ChannelDescriptor

某个 IO channel 的自描述对象，描述其输入输出能力、支持的数据类型、调用 ops、管理 ops、初始化/去初始化方法、回执类型与运行限制。

### 6.4 InteractionPolicy

交互表达决策策略，负责评估一个主体在当前人格、情绪、注意力、资源与上下文条件下是否想回应、用什么行为回应、通过什么模态回应。

### 6.5 TemporalDynamics

时间动力学层，负责将 tick、昼夜节律、无聊、恢复、情绪衰减、持续激动、习惯化与神经化学更新耦合起来。

### 6.6 Preconscious Path

前意识/非意识行为通路，负责在输入进入完整意识处理前，通过快速评估与低层优先机制生成候选动作，但仍需接入统一执行总线。

## 7. 当前问题陈述

### 7.1 交互表达问题

当前被动回复主要依赖单个 SEC 判断，缺少人格、注意力、疲劳、资源约束和交互欲的综合评估。

### 7.2 主动行为问题

当前主动行为主要依赖固定动作池和少量权重，无法充分表达一个主体性系统的复杂自发性。

### 7.3 时间建模问题

当前 tick 是运行节拍，不是完整的时间心理动力学机制。

### 7.4 行为与通道耦合问题

当前行为的外发路径仍然偏向固定逻辑，尚未做到“行为定义”和“channel 执行”解耦。

### 7.5 研究空缺问题

前意识/非意识行为与人格影响模块都缺乏现成的架构依据，必须先做研究映射。

### 7.6 行为可扩展性问题

行为池无法动态扩展，不利于后续复杂行为的增长和来源追踪。

### 7.7 长期记忆基础设施问题

长期记忆缺少数据库化存储与更强的检索能力，不利于复杂行为与交互决策复用记忆。

## 8. 正式需求

### R1. 交互表达评判逻辑增强

#### R1.1 目标

把“是否回复”从单一阈值判断升级为主体性交互策略判断。

#### R1.2 必须满足

1. 交互表达决策必须综合以下因素：
   - SEC 特征
   - 当前情绪状态
   - 当前人格投影
   - 当前注意力与认知负载
   - 疲劳与 allostasis 状态
   - 最近互动历史
   - 当前可用 channel 与行为能力
   - 当前动作成本与约束
2. 交互表达的结果不能只输出“回复/不回复”，还必须能输出：
   - 是否回应
   - 响应行为类型
   - 建议模态
   - 候选 channel
   - 触发原因与约束说明
3. 交互表达行为不应局限于文本回复，允许包含：
   - 文本消息
   - 文本转语音播放
   - 外部硬件驱动
   - 程序执行
   - 其他已注册合法行为
4. LLM 如参与决策，只能输出结构化提议，不得直接执行。

#### R1.3 验收标准

1. 存在独立的交互策略模块。
2. 该模块输出结构化决策结果，而不是单纯 bool。
3. 被动回复路径改为走统一动作提议接口。

### R2. 主动触发逻辑增强

#### R2.1 目标

把主动触发升级为多因子、可配置、可学习的策略评估过程。

#### R2.2 必须满足

1. 主动触发必须至少考虑：
   - 情感偏离
   - drive urgency
   - 人格特质投影
   - 神经化学状态
   - 注意力状态
   - 昼夜与时间动力学
   - 最近行为成败反馈
   - 动作成本与资源占用
2. 不同性格特质必须显式影响触发阈值与动作偏好。
3. 固定硬阈值与固定权重应迁移为可配置或可学习策略参数。

#### R2.3 验收标准

1. 现有主动行为路径不再只依赖单一固定权重组合。
2. 人格模块与神经化学门控能参与评分。
3. 输出仍为统一动作提议结构。

### R3. Tick 与时间动力学增强

#### R3.1 目标

把 tick 机制从调度循环升级为主体感受时间与慢变量更新的核心机制。

#### R3.2 必须满足

1. tick 必须参与以下状态的正规化更新：
   - boredom
   - 情绪自然衰减与恢复
   - 持续激动或持续压抑
   - habituation 与 novelty 衰减
   - 昼夜节律
   - 神经化学层的时间演化
2. 时间、内部状态和外部输入都必须能够影响神经化学层。
3. tick 设计必须与现有 affect、drive、thinking、memory 协同，而不是新建平行状态机。

#### R3.3 验收标准

1. 存在独立的时间动力学设计或模块。
2. boredom、恢复、情绪抹平至少具备明确更新模型。
3. neurochem 更新正式接入时间动力学层。

### R4. 主动输出 channel-agnostic 化

#### R4.1 目标

让具体输出行为由统一策略层和行为层决定，而不是写死到某个 channel。

#### R4.2 必须满足

1. 主动输出的最终执行 channel 不能写死为 QQ。
2. 行为选择必须可以参考：
   - 已注册 channel 能力
   - 当前情感状态
   - 人格状态
   - 神经化学状态
   - 输入上下文
   - 资源和权限限制
3. 行为定义与 channel 绑定必须通过结构化描述完成。

#### R4.3 验收标准

1. 外发行为通过 channel descriptor 与行为规格做匹配。
2. 行为执行层不再依赖单一固定 channel id。

### R5. 前意识/非意识行为路径模块

#### R5.1 目标

在意识层之前增加独立的数据路径与候选动作生成机制。

#### R5.2 必须满足

1. 该模块必须先以学术研究为依据再进入编码。
2. 必须先收集可靠、权威、可追溯的论文和综述来源。
3. 研究输出必须包括：
   - 来源目录
   - 核心观点摘要
   - 与 Helios 架构映射
   - 可实现的最小机制建议
4. 该模块产生的动作不能绕过统一策略和执行总线。

#### R5.3 验收标准

1. 代码实现前必须先存在研究文档。
2. 研究文档中的来源必须足够支撑模块边界。
3. 模块接入统一 ActionProposal 流程。

#### R5.4 Phase 2: 约束强制与运行边界

1. preconscious proposal 如声明 `internal_only`，planner/policy 层必须强制拒绝任何外部 channel 绑定。
2. preconscious path 的第一版 rollout 应保持为 regulation 未产出可执行候选时的 fallback，而不是抢占主动主路径。
3. preconscious proposal 的约束拒绝原因必须可追踪，便于后续回归和调试。

#### R5.5 Phase 3: 反馈闭环与可观测性

1. preconscious proposal 的执行结果必须进入统一 feedback 流程。
2. 系统必须能够记录最近一次 preconscious assessment、被拒绝原因或被接受理由，用于长期调试与分析。
3. 后续如引入 preconscious learning，只能基于统一 feedback 结果做受控、可衰减的评分调整，不得绕过现有 planner/executor/feedback 总线。

### R6. 人格影响模块

#### R6.1 目标

把人格从静态状态字段升级为显式影响投影模块。

#### R6.2 必须满足

1. 人格影响模块必须先完成研究收集再实现。
2. 模块至少要能输出对以下项目的偏置：
   - 交互欲
   - 主动触发阈值
   - 行为偏好
   - 风险容忍度
   - channel 偏好
   - 表达风格
3. 人格影响必须是独立模块，而不是散落到各评分函数中的常量乘法。

#### R6.3 验收标准

1. 存在独立人格影响投影接口。
2. 交互策略和主动策略都能消费人格偏置。

#### R6.4 Phase 2: 广域消费者接入

1. personality projection 不应只影响 interaction/regulation 两个模块，还应按依赖顺序逐步扩展到 internal thinking、temporal pressure、neurochemical gating 等评分面。
2. 新消费者必须读取结构化 projection bias surface，而不是重新散布 raw trait 常量。
3. 人格始终作为 slow prior，只能 bias 当前态评分，不得替代 affect、temporal、neurochem 和上下文约束。

#### R6.5 Phase 3: 可追踪影响与长期统计

1. 关键策略路径必须暴露 `personality_influence_trace` 或等价结构，说明本次决策受哪些人格 bias surface 影响。
2. 后续闭环记录中应支持汇总人格影响分布，用于审计、分析和长期稳定性验证。

### R7. 行为模块数据库化与动态扩展

#### R7.1 目标

建立可扩展、可追踪、可学习、可审计的行为注册表。

#### R7.2 必须满足

1. 行为定义必须迁移为数据库持久化。
2. 每个行为必须至少记录：
   - 行为 id
   - 行为名称
   - 描述
   - 参数 schema
   - 可用 channel
   - 冷却与代价
   - 适用条件
   - 来源
   - 版本
   - 激活状态
   - 学习统计
3. 允许 LLM 提议新行为，但必须经过：
   - 结构合法性校验
   - 能力/权限匹配
   - 审核或沙箱验证
   - 注册后才可执行
4. 行为的来源必须逐条记录，例如：
   - 图像观察
   - 搜索学习
   - 记忆联想
   - 推理生成
5. 行为结果必须进入运行闭环反馈。

#### R7.3 验收标准

1. 存在数据库化行为注册表。
2. 行为来源、执行结果、通道绑定可查询。
3. 新行为提议不会绕过注册与验证流程。

### R8. IO Channel 自描述与 ops 增强

#### R8.1 目标

让每个 channel 能以统一结构描述自身能力与操作接口。

#### R8.2 必须满足

1. 每个 channel 必须提供 descriptor，至少描述：
   - 输入数据类型
   - 输出数据类型
   - 数据格式
   - 解析方法说明
   - 可用 ops
   - 初始化方法
   - 去初始化方法
   - 健康检查/管理 ops
   - 输出调用函数集与格式说明
2. descriptor 必须可被系统查询。
3. channel ops 必须可被统一策略和执行层消费。

#### R8.3 验收标准

1. 存在统一 ChannelDescriptor 数据结构。
2. ChannelGateway 可查询注册 channel 的能力快照。

### R9. 记忆系统数据库化与 LLM 记忆架构增强

#### R9.1 目标

引入数据库后端，增强长期记忆保存、检索、压缩审计和 LLM 检索利用能力。

#### R9.2 必须满足

1. 记忆系统设计必须参考主流 LLM 记忆框架，并结合 Helios 当前四层记忆语义做整合。
2. 必须保留至少以下语义层：
   - working
   - episodic
   - semantic
   - autobiographical
3. 长期记忆必须具备数据库持久化能力。
4. 需要为后续引入向量检索、压缩审计、来源追踪和跨层检索预留接口。
5. 不得在第一版重构中破坏现有记忆闭环语义。

#### R9.3 验收标准

1. 存在数据库后端抽象与迁移方案。
2. 现有记忆层语义与新后端兼容。
3. 检索接口可扩展到更复杂的 LLM 记忆利用场景。

## 9. 横向约束

### 9.1 安全与权限

所有外部动作必须经过能力检查、权限检查与资源检查。

### 9.2 统一反馈

所有动作执行都必须产出结构化结果，供调节、记忆和分析系统消费。

### 9.3 兼容性

新架构必须尽量复用现有 `helios_main.py` 编排、`BehaviorExecutor` 执行队列和现有 memory 语义，而不是整体推倒重来。

### 9.4 可观测性

关键决策、行为来源、评分、约束、执行结果、通道回执必须可日志化和可追踪。

## 10. 依赖顺序

本轮需求存在明确依赖顺序：

1. 先冻结需求文档。
2. 再完成 low-level 设计。
3. 再输出任务拆分。
4. 再实现统一动作模型和 channel descriptor。
5. 再实现交互策略和主动策略增强。
6. 前意识与人格影响模块必须先完成研究收集。
7. 再推进行为 DB 和记忆 DB 重构。

## 11. 风险

1. 如果在没有统一动作抽象前直接增强 passive 和 active 逻辑，会形成更多分叉和重复实现。
2. 如果在没有 channel descriptor 前直接做多通道行为，会把行为与 channel 再次硬编码耦合。
3. 如果允许 LLM 直接添加并执行行为，会带来不可控风险和不可审计执行路径。
4. 如果在没有研究前置的情况下实现前意识行为和人格影响模块，容易陷入伪理论或随意设计。
