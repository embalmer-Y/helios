# 项目开发流程指南

## 核心原则

本项目采用结构化的 Spec/Task 流程进行开发。所有功能开发、Bug 修复、重构等工作都必须遵循此流程。

---

## 目录结构

```
.specs/
├── feature-name/
│   ├── requirements.md    # 需求文档
│   ├── design.md          # 设计文档
│   └── tasks.md           # 任务清单及状态
```

---

## 流程阶段

### 阶段一：Inception（需求与设计）

1. **Requirements（需求）**
   - 明确功能目标、用户场景、约束条件
   - 列出验收标准（Acceptance Criteria）
   - 记录在 `requirements.md`

2. **Design（设计）**
   - 技术方案选型、架构设计
   - 接口定义、数据结构设计
   - 记录在 `design.md`

### 阶段二：Construction（实施）

3. **Tasks（任务拆分与执行）**
   - 将设计拆分为可独立执行的原子任务
   - 每个任务有明确的完成标准
   - 按顺序逐个执行
   - 记录在 `tasks.md`

### 阶段三：Verification（验证）

4. 每个 task 完成后进行验证
5. 所有 tasks 完成后进行整体验收

---

## Tasks 文件格式

```markdown
# Tasks - [功能名称]

## 状态说明
- [ ] pending - 待执行
- [~] in_progress - 执行中
- [x] completed - 已完成
- [!] blocked - 被阻塞

## 任务列表

### Task 1: [任务标题]
- 状态: completed
- 描述: [具体要做什么]
- 完成标准: [怎样算完成]
- 备注: [执行过程中的记录]

### Task 2: [任务标题]
- 状态: pending
- 描述: [具体要做什么]
- 完成标准: [怎样算完成]

...
```

## 执行规则

### 开始工作前

1. 读取 `.specs/` 目录，了解当前所有进行中的 spec
2. 找到当前 spec 的 `tasks.md`
3. 确认上一个已完成的 task，定位下一个待执行的 task

### 执行 Task 时

1. 将目标 task 状态更新为 `in_progress`
2. 严格按照 task 描述和完成标准执行
3. 一次只执行一个 task，不要跳跃
4. 如果遇到阻塞，标记为 `blocked` 并记录原因
5. 完成后将状态更新为 `completed`，记录备注

### 完成 Task 后

1. 验证完成标准是否满足
2. 更新 `tasks.md` 中的状态和备注
3. 如果有后续依赖的 task，检查是否可以继续

---

## 沟通规则

- 每次开始工作时，先报告当前进度（第几个 task）
- 遇到设计文档与实际情况冲突时，先询问再行动
- 完成一个 task 后简要汇报结果
- 不确定的决策点，列出选项让用户选择

---

## 质量标准

- 代码变更必须保证现有功能不被破坏
- 遵循项目已有的代码风格和约定
- 新增功能需要有对应的错误处理
- 配置项不硬编码，使用变量或配置文件

---

## 恢复上下文

如果对话中断或上下文丢失：

1. 读取 `.specs/` 下的 spec 文件
2. 查看 `tasks.md` 中各 task 的状态
3. 找到最后一个 `completed` 的 task
4. 从下一个 `pending` 的 task 继续执行
5. 不需要重复已完成的工作
