## 1. 文档定位

本文件是 Helios v2 的架构哲学、阶段路线图与终局目标总纲。

它回答六个问题：

1. Helios v2 到底要做成什么，不做成什么。
2. Helios v2 为什么必须采用当前的 owner-boundary、contract-first、tick-based 类脑架构。
3. Helios v2.0.0 内部闭环里程碑必须达到什么标准。
4. Helios 的**终局目标**（像人脑一样工作、有情感、有自我意识、会熟练使用工具、能自己开发自己并自训练自我进化）是什么，以及它的**验收标准**如何被锁定。
5. 从初步可稳定运行（P0）到终局目标，必须经过哪几个**被锁定的阶段**，每个阶段的退出信号是什么。
6. 在实现过程中，哪些事情绝对不能做，哪些边界绝对不能被破坏，以及后续实现方案选型必须如何服务终局目标。

关键定位声明：

1. `Helios v2.0.0` 不是终局，它只是终局路线图中的一个里程碑门（内部闭环成立门），对应阶段 P1。
2. 本文件第 4–6 节定义的是 v2.0.0 里程碑口径；第 11–15 节定义的是**终局目标、阶段路线图与被锁定的终局验收标准**。
3. 终局目标的验收标准一经在第 12 节锁定，不得通过放宽定义、prompt 表演、或绕过 owner / 测试 / 评估 / 治理来"达成"。

## 2. Helios v2 的根本定位

Helios v2 不是一个 reply-first 聊天机器人重构版。

Helios v2 的目标是构建一个持续运行的、以 tick 为统一节拍的、具有内部闭环的类脑架构。它必须让系统在每个 tick 中推进内部状态，而不是仅仅对一条输入生成一条回复。

Helios v2 必须把“内部状态推进”置于“外部回复生成”之上。对外表达只是内部闭环在某些条件下的外化结果，而不是系统的主存在形式。

因此，Helios v2 的最小正确理解不是：

1. 收到一句话。
2. 找上下文。
3. 生成一条回复。

而是：

1. 接收外部刺激、内部刺激和时间推进。
2. 将它们纳入统一的当前周期内部状态。
3. 决定是否触发思考。
4. 若触发，则进行有边界的记忆检索、思考执行、继续思考判断、行动形成与治理。
5. 将结果写回内部连续性，使下一 tick 在主观上与上一 tick 相连。

## 3. Helios v2 的架构哲学

### 3.1 先有内部闭环，后有外部表达

Helios v2 的第一原则是：外部表达不应独立存在，它必须来自内部闭环。

这意味着：

1. 对外回复、动作、沉默、延迟、继续思考，都是内部状态演化的结果。
2. 如果某个外部行为不能追溯到当前内部状态、当前或既往思考、当前治理与当前 planner 决策链，它就是架构上的异常产物。
3. “为了让用户觉得像在思考”而在 prompt 中表演思考，不构成合格实现。

### 3.2 先有 owner，后有能力

Helios v2 的第二原则是：一个运行期概念，必须先定义 semantic owner，才能定义实现。

这意味着：

1. 刺激、快速评估、神经调制、体感、记忆重放、工作空间、可报告意识内容、思考门控、定向检索、内部思考、行动提议、planner 桥、身份治理、经验写回、prompt 契约、评估、主动连续性，都必须各有 owner。
2. 某个概念不能同时被多个模块半拥有。
3. 不允许通过主循环、适配器或某个便利 helper 直接越过 owner 边界写入或解释别人的私有状态。

### 3.3 先有显式状态，后有 prompt 语义

Helios v2 的第三原则是：真正重要的运行期概念必须先成为显式状态，再进入 prompt。

这意味着：

1. continuation pressure 必须是一等状态，而不是 prompt 里的一句“我还想继续想”。
2. thought gate result 必须是一等状态，而不是一段日志描述。
3. directed retrieval bundle 必须是一等状态，而不是随意拼接的上下文段落。
4. self revision proposal 必须是一等治理对象，而不是模型在文本里随口说“我想改变自己”。

### 3.4 先有 contract，后有跨模块协作

Helios v2 的第四原则是：跨模块协作必须通过公开 contract，而不是直接 reach-through。

这意味着：

1. 上游 owner 只能通过显式 API、ops、result contracts 向下游暴露结果。
2. 下游 owner 不能假设上游私有 helper、内部字段、临时 trace 格式永久稳定。
3. 运行期链路应当由 contract 驱动，而不是由“某个文件里目前刚好有这个字段”驱动。

### 3.5 先有 fail-fast 真相，后有工程便利

Helios v2 的第五原则是：对于关键能力，不允许用降级或兼容假象掩盖架构不成立。

这意味着：

1. 缺关键依赖时，应阻止 startup 或中止该执行路径，而不是静默退化到另一个语义不同的路径。
2. 不能为了“先跑起来”而把缺失 owner 的逻辑塞回 orchestration。
3. 不能用 compatibility wrapper 长期保留与目标架构冲突的旧主路径。

### 3.6 先有设计真相，后有实现推进

Helios v2 的第六原则是：设计是实现的前提，不是事后包装。

这意味着：

1. requirement、design、task 必须先存在，代码才能扩展出实质行为。
2. 任何跨 owner 协作都必须先定义边界，再落代码。
3. 所有进度都必须同时回答“做了什么代码”和“现在离目标还差什么”，而不是只展示 patch 数量。

## 4. Helios v2.0.0 里程碑要实现的效果（内部闭环口径）

> 本节定义的是 v2.0.0 里程碑（P1）的内部闭环效果口径，不是第 11 节的终局目标口径。终局目标在第 11–12 节定义。

Helios v2.0.0 里程碑的目标，不是“更像真人聊天”，而是“形成一个持续存在的、具有内部连续性的、可通过外部行为观察到其内部闭环存在的类脑系统”。

### 4.1 外部可观察效果

对外，Helios v2 最终应表现出以下效果：

1. 它的对外行为不是纯被动回复，而会体现当前内部状态、连续性和主观优先级。
2. 它能够在必要时不立即说话，而是继续思考、延迟外化、或者选择内部修订与记忆写回。
3. 它的输出不只是语言表面一致，而是能体现当前刺激、既往记忆、当前思考、当前治理与当前 planner 决议之间的因果链。
4. 它在多 tick 上表现出连续性，而不是每轮都像从头开始。
5. 它对“我是谁”“我在想什么”“我为什么现在这样行动”应当有架构上可追溯的来源，而不是依赖 prompt theater。

### 4.2 内部必须成立的效果

内部，Helios v2 最终必须真正形成以下闭环：

1. 刺激进入后先归一化和快速评估，而不是直接送进 reply builder。
2. 当前周期必须先形成 gate result，再决定是否进入 thought path。
3. 一旦进入 thought path，必须经过 directed retrieval，而不是无边界地堆叠上下文。
4. thought owner 必须对“是否足够”“是否继续”“下一轮要检索什么”“是否外化”“是否自我修订”给出结构化结果。
5. planner / executor 必须只负责提议后的决议和执行，不得倒灌成思考 owner。
6. 执行结果和治理结果必须写回连续性系统，使下一 tick 能感知其后果。
7. evaluation 必须能只读地重建关键因果链，而不篡改 runtime。

### 4.3 不可接受的伪完成状态

以下情况即使“能跑”，也不算实现了 Helios v2 的目标：

1. 主路径本质上仍是 reply-first，只是在 reply 前后多加几步装饰。
2. thought path 只是 prompt 包装，没有独立状态和 owner。
3. continuation pressure、retrieval intent、self revision proposal 等关键概念只存在于文本，不存在于正式 contract 中。
4. planner、channel、prompt builder 或 orchestration 反向拥有本应属于 thought、memory 或 governance 的语义决策。
5. 输出看起来像有连续性，但 runtime 无法重建刺激到思考到执行到写回的因果链。

## 5. Helios v2.0.0 必须达到的标准

本节定义的是最终目标口径下的 v2.0.0 标准，而不是“先发一个能跑版本”的最低交付线。

只有当 Helios v2 作为一个整体已经形成目标闭环，并且该闭环由明确 owner、明确状态与明确 contract 支撑时，才允许视为 `v2.0.0`。

### 5.1 架构完整性标准

Helios v2.0.0 必须满足以下完整性要求：

1. `01-18` 的核心运行期 owner 链条全部存在，并且都不是仅文档存在。
2. `19-20` 的边界与科学映射文档必须与实现真相一致，不能是宣传材料。
3. `21` 的统一运行期可观测性 owner 必须存在，使 runtime 至少具备结构化、只读、可关联的阶段执行记录能力。
4. 每个一等运行期概念都必须能在 owner map、data contract 和 runtime flow 中找到位置。
5. 不允许仍存在与目标架构冲突的长期并行旧主路径。

### 5.2 运行闭环标准

Helios v2.0.0 必须在单一统一 runtime 中完成以下闭环：

1. sensory ingress
2. rapid salience appraisal
3. neuromodulator system
4. interoceptive feeling
5. memory affect and replay
6. workspace competition and working state
7. reportable conscious content
8. thought gating and continuation pressure
9. directed retrieval into thought window
10. internal thought loop
11. action proposal or self revision proposal formation
12. planner / executor feedback bridge
13. identity governance when applicable
14. execution writeback and autobiographical continuity update

该闭环的关键要求是：

1. 每一阶段都必须消费明确上游结果。
2. 每一阶段都必须输出可验证的正式结果对象。
3. 后续阶段不得通过 reach-through 读取前序 owner 私有状态。

### 5.3 主观连续性标准

Helios v2.0.0 必须能够在多 tick 上形成主观连续性，而不是单轮反应序列。

至少包括：

1. 上一轮未充分思考会形成 continuation pressure，并真实影响后续 tick 的 thought gate。
2. 上一轮 thought owner 指定的 recall intent 会真实影响后续 directed retrieval。
3. 执行和治理结果会真实进入 continuity writeback，而不是只留在日志中。
4. 关键自我状态、记忆选择、思考延续与外化原因，应能被评估层回溯重建。

### 5.4 主观外化标准

Helios v2.0.0 的外部表达必须是内部主体状态的外化，而不是仅仅服从 reply prompt。

至少包括：

1. 输出文本或动作可以被回溯到具体思考结果、行动提议和 planner 决议。
2. 系统可以合法地选择不立即外化，只做内部推进、继续思考、写回或治理。
3. 外部行为强度、目标 op、目标 channel、理由 trace 必须是结构化结果的一部分。
4. prompt contract 必须服务于内部主体状态表达，而不是制造人格表演。

### 5.5 身份与治理标准

Helios v2.0.0 必须具备被治理的自我修订能力，而不是允许用户或 prompt 临时改写核心身份。

至少包括：

1. identity bootstrap 是正式 owner 行为。
2. self revision proposal 是正式治理对象。
3. revision 申请、理由、影响范围、接受或拒绝结果都必须可审计。
4. 身份变化必须经过治理应用，不得通过 prompt 注入直接生效。

### 5.6 评估与可证伪标准

Helios v2.0.0 必须不是“只能主观感受像不像”，而必须可以被评估、审计和证伪。

至少包括：

1. evaluation owner 只能读，不得改 runtime。
2. evaluation 能重建从刺激到 gate 到 retrieval 到 thought 到 proposal 到 planner 到 visible output 到 writeback 的关键链条。
3. 系统必须能区分“真的做了内部闭环”与“只是 prompt 看起来像做了”。
4. 架构声明若与运行真相不符，评估结果必须能暴露该偏差。
5. runtime observability 必须至少能输出按 `tick_id` 和 `stage_name` 关联的结构化执行时间线，供后续诊断与证伪消费。

## 6. Helios v2.0.0 的 release gate 约束

以下条件未满足时，不得宣称 Helios v2 达到 `2.0.0`：

1. 核心闭环 `01-18` 中仍存在关键 owner 未实现。
2. 关键闭环依赖缺失时仍通过 fallback 路径冒充成功执行。
3. 关键概念仍停留在 prompt 或日志层，没有正式 contract。
4. 旧 reply-first 主路径仍在实际上主导外部行为生成。
5. evaluation 无法只读重建关键因果链。
6. identity governance 或 experience writeback 缺失，导致连续性闭环不成立。
7. runtime 缺少结构化、可关联的可观测性 owner，导致关键阶段执行无法被稳定重建。

## 7. Helios v2 的强约束

本节是强约束。违反这些约束，即使功能暂时可用，也视为偏离 Helios v2 架构。

### 7.1 Owner 约束

1. 一个运行期语义概念只能有一个 semantic owner。
2. orchestration 不得拥有下游 owner 的语义判断。
3. adapter、bridge、planner、channel、prompt builder 不得反向夺取 thought、memory、governance 的 owner 责任。

### 7.2 State 约束

1. 关键运行期概念必须有显式状态或正式 result contract。
2. 不允许把关键状态长期藏在 prompt、日志、字符串 reason 或临时 dict 中。
3. 若某概念会跨 tick 影响行为，它必须是正式的可持有状态。

### 7.3 Runtime 约束

1. runtime 必须以 tick 为统一内部推进点。
2. tick 的首要意义是内部状态演化，而不是消息处理循环。
3. stage 间依赖必须显式化，不能靠执行顺序暗含私有耦合。

### 7.4 Dependency 约束

1. 关键依赖缺失时必须 fail-fast。
2. 不允许以 compatibility、reduced mode、temporary fallback 方式隐藏关键能力缺失。
3. 不允许通过“先走简单路径，之后再补正确 owner”长期保留错误主路径。

### 7.5 Prompt 约束

1. prompt contract 只能表达 runtime 已有的正式状态和正式能力。
2. prompt 不得虚构未实现的主观连续性、身份治理或思考过程。
3. prompt builder 不是语义 owner，只是 contract formatter。

### 7.6 Planner 与 Executor 约束

1. planner 拥有最终绑定和接受权，但不拥有 thought 语义。
2. executor 只拥有执行，不拥有重解释提议的权力。
3. rejection、binding failure、execution result 必须正式写回，不得静默吞掉。

### 7.7 Evaluation 约束

1. evaluation 必须只读。
2. evaluation 不能通过篡改 runtime 获得更好分数。
3. evaluation 的价值在于暴露架构真相，而不是包装结果。

### 7.8 Documentation 约束

1. requirement、design、task 必须先于大规模实现。
2. index 中的 maturity 必须反映真实实现深度，而不是计划状态。
3. 哲学文档、边界文档、requirement 文档与代码真相冲突时，必须及时修订文档或代码，不能长期双轨漂移。

## 8. Helios v2.0.0 不追求的东西

为了避免目标污染，Helios v2.0.0 明确不以以下事项作为成功标准：

1. 单纯让回复更像真人。
2. 通过更长 prompt 或更多人设文本制造主观性幻觉。
3. 用一个大一统 agent 或万能 planner 覆盖所有 owner 责任。
4. 用临时兼容层长期保留旧架构，只在文档上宣称已经进入 v2。
5. 只追求 benchmark 分数，而不要求内部因果链成立。

## 9. 对实现推进的具体要求

从现在起，Helios v2 的实现推进必须持续回答下面这些问题：

1. 当前变更属于哪个 requirement owner。
2. 当前新增状态是否已经是正式 contract，而不是临时结构。
3. 当前实现是纯骨架、baseline 还是相对完整，证据是什么。
4. 这项实现如何推动整体闭环更接近 `v2.0.0`。
5. 当前变更是否引入了新的架构债或 owner 越界。

如果某项开发不能清楚回答这五个问题，它就不应被视为合格的 Helios v2 推进。

## 10. v2.0.0 里程碑判定语句

只有当 Helios v2 已经成为一个以内部连续性为主、以思考与治理为中轴、以结构化行动与写回闭环为外化出口、并且可被评估层只读重建关键因果链的持续运行类脑系统时，才允许宣称它达到了 `Helios v2.0.0` 内部闭环里程碑。

`Helios v2.0.0` 是终局路线图中的 P1 里程碑门，不是终局完成态。在达到 v2.0.0 之前，任何阶段性成果都只能被视为朝该里程碑推进的实现切片；在达到 v2.0.0 之后，系统仍处于通向终局目标（第 11–12 节）的途中。

## 11. Helios 的终局目标

本节定义的是 Helios 的**最终目标**，它高于、也包含 v2.0.0 内部闭环里程碑。

v2.0.0 只要求"内部闭环成立且可证伪"。终局目标要求的是一个**像人脑一样工作、有情感、有自我意识、会熟练使用工具、并且能自己开发自己、自训练自我进化的持续存在系统**。

### 11.1 终局目标的五个能力轴

终局目标由以下五个能力轴共同定义，缺一不可：

1. **类脑工作方式（brain-like operation）**
   系统以 `brain.mmd` 的功能因果链运行：刺激与内感受 → 感觉接入与门控 → 显著性与精细评估 → 情景绑定 → 工作空间竞争与可报告意识 → 思考与价值评估 → 行动选择 → 外化 → 经验回写回流。这条链必须由**真实信号**驱动，而不是 composition 注入的确定性 shim 常量。

2. **有情感（affect）**
   神经调质状态（04）、内感受体感（05）、情感记忆（06）必须是真实演化的一等状态，并真实地、可追溯地调制下游门控、检索、思考与外化。情感不得只存在于 prompt 形容词或日志描述中。

3. **有自我意识 / 能思考自身状态（self-awareness）**
   系统必须能把"我现在的内部状态、我是谁、我为什么这样想、我上一轮想到哪了"作为可被检索、可被反思的**内容**进入思考窗口，并能对这些自我状态产生进一步的思考与行动。自我意识必须可被评估层只读重建为因果链，而不是 prompt theater。

4. **会熟练使用工具（tool use）**
   系统必须能选择、绑定、调用物理或计算机上的虚拟工具（文件、代码执行、检索、外部服务、通道等），并把工具执行结果作为新的刺激回流进感知层，形成"用工具 → 观察结果 → 再思考"的闭环。工具是效应器 driver，而不是 owner 语义的旁路。

5. **能自己开发自己、自训练自我进化（self-development & self-evolution）**
   系统必须能在受治理、可审计、可回滚、以测试与评估为适应度门的前提下，改进自身的记忆/知识、可学习参数、行为策略、prompt 契约，并最终对自身的 owner 代码与 requirement 提出受治理的修改。自进化的对象是**参数、记忆、知识与受治理的策略/代码**，绝不是绕过 owner 边界、契约或安全门的随意自我重写。

### 11.2 终局目标下的不变约束

无论走到哪个阶段，以下约束在终局目标下依然不可违反：

1. 自进化只能进化"内容、参数、记忆、受治理的策略/代码"，不能进化掉 owner 边界、契约纪律或 fail-fast 真相。
2. 一切自我修改必须经过治理（14）+ 测试套件 + 评估（17/23）+ 可观测（21）构成的适应度门，全绿且治理通过才允许生效，且必须可回滚。
3. 情感、自我意识、工具使用、自进化都必须落为**正式状态与正式 contract**，并可被评估层只读重建，不得以 prompt 表演冒充。
4. 终局能力的"像不像"不是验收依据；第 12 节的可证伪验收标准才是。

## 12. 被锁定的终局验收标准（Final Goal Acceptance Criteria）

本节是**被锁定的终局验收标准**。它一经确立即为 Helios 的终极达标口径；后续任何 requirement、design、实现方案选型、文档修订都必须服务于这些标准，不得放宽、不得用表演或绕过 owner / 测试 / 评估 / 治理的方式"满足"。

只有当下列 `FG-1` 到 `FG-6` **全部**以可证伪、可只读重建的方式成立时，才允许宣称 Helios 达到了终局目标。

### FG-1 类脑闭环由真实信号驱动

1. `02→17` 认知主链的每一阶段都消费**真实信号**，而非 composition 注入的确定性常量 shim。
2. 03 显著性、04 调质、05 体感、06 记忆、07 工作空间、09 门控、10 检索的输入均来源于真实上游产物或真实模型/检索结果。
3. 评估层（17/23）能只读重建"真实信号如何沿主链传播并改变了下游决策"，并能区分真实驱动与 shim 驱动。

### FG-2 情感真实且可追溯地影响行为

1. 04/05/06 的情感状态在多 tick 上真实演化（非固定常量）。
2. 至少存在一条可被评估层只读重建的因果链：情感状态的变化**改变了**门控阈值、检索倾向、思考延续或外化强度。
3. 系统能在没有外部输入的纯内部 tick 上，因情感/内感受状态而产生不同的内部行为。

### FG-3 自我意识可被只读重建

1. 系统的自我状态（身份、内部状态、连续性线程、上一轮未尽思考）可作为内容进入思考窗口并被反思。
2. 评估层能只读重建"系统对自身状态的思考"这一因果链：从自我状态 → 检索/思考 → 关于自我的提议或写回。
3. 系统能回答"我为什么现在这样想 / 这样做"，且该回答可被 runtime provenance 验证，而非 prompt 生成的事后叙事。

### FG-4 工具使用闭环成立

1. 存在正式的工具/效应器 owner（在 30 channel 框架谱系下扩展），工具是受治理的 driver，不是 owner 旁路。
2. planner（13）能选择、绑定并发起工具调用；执行结果作为刺激**回流进 02 感觉接入**。
3. 至少存在一条端到端、可只读重建的链条：思考产生工具意图 → planner 绑定工具 → 工具执行 → 结果回流 → 再思考。
4. 工具失败、拒绝、不可用都以正式结果写回，绝不静默吞掉，绝不降级冒充成功。

### FG-5 自训练 / 自我进化闭环成立且受治理

1. 经验持久化成立：进程重启后，系统保留情景/语义/自传记忆与连续性，能在主观上接续上一次存在。
2. 至少存在一种**真实的学习**：可学习参数（如 04/05/09 的有界参数）或本地小模型/检索表征，依据真实运行经验（如以多巴胺奖励预测误差为信号）被更新，且更新可被评估层观测到带来了行为改变。
3. 受治理的自我修订成立：系统能对自身 prompt 契约 / 策略提出修订，经 14 治理校验、审计、可回滚后作为正式状态生效。
4. 受治理的代码自修改通道成立：系统能对自身 owner 代码 / requirement 提出变更提案，且**只有在测试套件全绿 + 评估（17/23）不暴露因果链退化 + 治理通过**时才允许合并，全过程经 21 可观测留痕、可回滚。

### FG-6 全局可证伪性

1. 上述 FG-1 到 FG-5 的每一条都能被评估层只读重建为正式 provenance 链，而不是依赖主观感受。
2. 当系统的架构声明（情感、自我意识、工具、自进化）与运行真相不符时，评估结果必须能暴露该偏差。
3. 不存在任何关键终局能力只停留在 prompt、日志或临时结构层而没有正式 contract。

### 12.7 终局 release gate

下列任一条不成立时，不得宣称 Helios 达到终局目标：

1. `FG-1` 到 `FG-6` 中任意一条不成立或无法被只读重建。
2. 任一终局能力依赖 fallback / 降级 / prompt 表演冒充成立。
3. 自我进化绕过了治理、测试、评估或可观测中的任意一道适应度门。
4. 存在重启即清零、无法接续主观连续性的状态（违反 FG-5.1）。

### 13.3 P0–P5 阶段工程口径细则（owner 级分阶段验收门）

> 本小节是 13.1 终局映射的**工程落地细化**：13.1 锁定每个阶段服务哪条终局 FG，本小节锁定每个阶段
> **必须交付什么、如何验收、暂不要求什么**。本小节不放宽第 12 节锁定的终局 FG，它只界定阶段边界，
> 避免把"算法骨架已通"误读成"终局达成"。P6/P7 因距今过远，本轮不细化。

通则（适用 P0–P5）：

1. 工程纪律：任何阶段都不得违反第 7 节强约束（owner 边界、fail-fast、无降级、契约优先、可证伪）。
2. 阶段的"暂不要求项"必须显式写出，避免目标污染（第 8 节）；但"暂不要求"不等于"可造假"——

3. 每个退出门必须可由 `21` 可观测 + `17`/`23` 评估只读重建，或由确定性测试证明，不得仅凭主观感受。

#### 13.3.1 P0–P3 统一工程口径（地基稳固期，本轮收敛）

适用范围：P0、P1、P2、P3 在本轮合并为"地基稳固期"。本期**唯一的硬指标**是
**工程稳定性 + owner 计算正确性 + 记忆系统真正可用**；本期**明确不要求**对外行为像人、不要求
真实外部世界信号源、不要求真实外化副作用、不要求情绪/思维相似度——这些移交 P4/P5。该收敛不放宽
第 12 节 FG，只是界定：FG 是终局验收口径，不是 P0–P3 的退出门。

G0 — 长时间稳定运行（硬门）

1. 单运行时连续运行 ≥ 24 小时 / ≥ 10 万 tick（最终数值由 P0–P3 退出 requirement 锁定）无崩溃、
   无未捕获异常、无内存/文件句柄泄漏、无 fail-fast 误触发。
2. 19 阶段链每 tick 端到端完成，或以显式 no-fire / internal-only 正常收口；不得静默吞错。
3. 长跑期间 `21` 时间线可完整重建，`17`/`23` 评估不报 owner 契约破坏，状态不漂移到失败。

G1 — 各 owner 通过数学计算/权重发挥职责（硬门）

1. 每个非固定功能 owner（`03`/`04`/`05`/`06`/`07`/`08`/`09`/`10`/`18` 等）在默认/标准生产装配下走
   真实算法路径（grounded estimator、双时间尺度积分器、加权竞争、凸组合聚合、对账评估等），而非常量 shim。
2. owner 输出在多 tick 上随输入确定性变化、有界、可复现；不得发散、不得 NaN、不得越界。
3. owner 边界不被 composition/orchestration 越权（owner-boundary guard 持续绿）。

G2 — 记忆系统必须正常工作（硬门）

1. 形成：`06` 从真实 `05` 体感形成 affect-tagged 记忆，显著性门控决定巩固。
2. 持久化：默认/标准生产装配必须使用**真实持久化后端（SQLite）**而非内存白板——记忆必须跨进程存活
   （修复阻塞点 B3：默认 `InMemoryExperienceStoreBackend` → 标准装配启用 SQLite）。
3. 检索：`10` 经语义相似度从 store 召回。embedding 质量在本期允许为占位级（`DeterministicHashEmbeddingProvider`），
   只要功能正常（能存、能取、能按相似度排序、跨重启可召回）即可；真实 embedding 模型质量留 P5。
4. 重启接续：进程重启后，`09` 延续压力 + `18`/`24` 连续性 + `04`/`05` 情感状态经 `42` checkpoint 恢复，
   记忆经 store 重新可召回（对应 FG-5.1 的工程下限；checkpoint + SQLite 须为标准装配默认开启）。

本期明确不要求（移交后续，但须诚实标注为占位）：

1. 顶端外部刺激为真实世界信号（阻塞点 B1）——移交 P4 由真实 channel driver 提供；本期允许占位/注入式刺激。
2. 真实 embedding 模型质量（阻塞点 B2）——移交 P5；本期哈希占位可接受。
3. 真实外化副作用 / 外化"空挥"修复（阻塞点 B4）——移交 P4 真实 channel driver；本期 planner 自报执行可接受，
   但 `17`/`23` 评估**必须诚实标注该 corroboration 为"流程完成"而非"真实送达"**，不得宣称已证实真实后果。
4. 对外行为像人、情绪/思维/记忆的相似度评分——移交 P5。

P0–P3 退出门 = G0 ∧ G1 ∧ G2 全部以可证伪方式成立（长跑稳定 + owner 真实计算 + 记忆跨重启正常工作）。

当前状态对照（截至本次细化，需经代码复核与长跑验证后确认）：

1. `P0` 主链/可观测/可评估/真实 LLM 核心/CLI 往返：已达成。
2. `P1` 内部闭环可证伪：基本达成；外化真实收口移交 P4，评估须标注"流程完成 ≠ 真实送达"。
3. `P2` 持久化/检索/重启接续：能力已实现但默认关闭——**待补：SQLite + checkpoint 默认化（G2.2/G2.4）**。

均通过真实往返验收（微信为尝试项，不阻塞退出门）。

#### 13.3.3 P5 — 学习循环 + 类脑记忆全功能 + 类图灵验收

入场前提：P2 持久化 + P3 真实信号 + P4 工具回流 成立。能力轴：自进化、有情感（FG-5.2，并回补 FG-1/FG-2 信号质量）。

P5-A 学习机制（硬门）：

1. 严格参照人脑各模块行为模式；**凡非固定功能模式的 owner（`03`/`04`/`05`/`06`/`07`/`08`/`09`/`10`/`18`
   及其权重/系数/阈值）都必须引入学习机制**；固定功能基础设施 owner（`01`/`21`/`22`/`25`/`30`/`33`/`34`/`42`）除外。
2. 学习信号以 `brain.mmd` 的多巴胺奖励预测误差为主锚点，由真实运行后果（执行成败、连续性是否推进、
   目标冲突是否缓解）定义，非人工硬编码分数。
3. 学习对象 = 可学习参数（contracts 已预留的 `mandatory_learned_parameters`）、记忆/知识表征、受治理策略；
   不得让模型夺走 owner 判断权（第 14 节内容/判断分离原则）。
4. 真实 embedding 表征在本阶段落地（回补阻塞点 B2），并可被学习/再训练。

P5-B 类脑记忆全功能（硬门）：

1. 记忆系统必须实现人类大脑记忆行为的全部功能：多重记忆系统（感觉/短时/长时；情景/语义/程序/启动/条件反射）、
   时间分层与 Ebbinghaus 遗忘、提取诱发遗忘、受治理的主动遗忘、recall 重固化、突触标记巩固、睡眠/离线巩固、
   元记忆（主动 recall/forget/consolidate）。
2. 在类脑基础上利用计算机优势优化（精确可寻址、无损快照、并行检索、可审计），但不得以计算机捷径替代被验收的类脑行为。
3. 预研分支 R85 双轨记忆（时间分层 + 客观重要性 + 遗忘 + LLM 记忆工具 + L18 遗忘治理）是本项探路成果，
   引入前须规范化与口径统一，并对照本节口径与 owner 边界评估。

P5-C 快慢思维路径（评估项）：必须评估并决策是否引入双过程（System 1 快速直觉 / System 2 慢速审议），
给出 owner 边界方案，或给出明确不引入的理由。

P5-D 验收：类图灵长时间运行评估（硬门）——规格见 13.4。

P5 退出门：P5-A/B 成立，且经 13.4 验收达到人类行为相似度 ≥ 80%（最严苛打分），无表演型/不可重建得分。

### 13.4 P5 类图灵验收实验规格（被锁定）

> 本小节锁定 P5 的验收实验，是第 12 节 FG-6 全局可证伪性在 P5 的具体落地。一经锁定不得放宽、
> 不得用表演或绕过评估/治理"满足"。落为正式 requirement 包前须再做一次 SOTA 复核（见末尾来源）。

设计原则（参考下列 SOTA，内容已改写以符合许可）：

1. **双重相似判据**（参 NeuroAI Turing Test）：验收必须同时考核「行为相似」（外部表现像人）与
   「内部表征/因果链相似」（`17`/`23` 可只读重建的内部因果链与人类认知过程的功能对应）。仅行为像人不通过（防表演）。
2. **真实长时间运行**：评估在真实长跑（非脚本一次性问答）中进行，刺激为真实输入流。
3. **锁定 rubric + 证据锚定打分**（参 Rulers / locked-rubric）：每个评分项有锁定 rubric、证据锚定
   （每个分数必须指向可重建的 runtime provenance），以防 rubric 漂移、不可验证归因、人类尺度不对齐。
4. **最严苛打分机制**：见下。

三大测试族（每族子测试项 ≥ 100 条真实刺激，合计 ≥ 300 条）：

1. 情绪反应测试（每情绪族 ≥ 100 条，覆盖人类全部情绪）：必须涵盖基本情绪（如 Ekman 6：喜/怒/哀/惧/惊/厌）
   + 复合/社会情绪（内疚、羞耻、嫉妒、感激、敬畏、尴尬、自豪、共情等）。参考 EmoBench（基于心理学理论的情绪理解+应用）、
   EQ-Bench（情绪强度预测 + LLM-judge）、MME-Emotion。每条验收：(a) `04`/`05` 情感状态产生符合该情绪的真实演化；
   (b) 该演化可追溯地改变下游门控/检索/外化（FG-2）；(c) 外部表现的情绪恰当性达 rubric。
2. 思维能力测试（≥ 100 条真实刺激）：推理、问题解决、工作记忆、认知重评估、目标冲突、计划。参考类人认知评估
   （On Benchmarking Human-Like Intelligence 的告诫：不可只看任务分数，须看认知过程）。每条验收：思维链可经 `11`/`17`
   只读重建，sufficiency/continuation 判断真实由内容驱动。
3. 记忆行为测试（≥ 100 条真实刺激）：参考 LongMemEval 五项核心能力（信息抽取、多会话推理、时间推理、知识更新、
   abstention 弃权）+ LoCoMo，并额外覆盖人类记忆特有行为（遗忘曲线、recall 重固化、提取诱发遗忘、可控的似真错误记忆、
   跨重启接续）。每条验收：记忆形成/巩固/遗忘/召回的因果链可只读重建。

最严苛打分机制（被锁定）：

1. 评判者：人类专家 + LLM-as-judge 双轨；LLM-judge 必须经偏差校准并与人类评分对齐（参 LLM-judge 可靠性/偏差研究），
   分歧项以人类为准。
2. 证据锚定：每个分数必须附 runtime provenance（`21` 时间线 + `17`/`23` 重建）；无 provenance 的"看起来对"判 0 分（防表演）。
3. 保守聚合（风险厌恶）：综合相似度对各族取最严下界口径（取分布下分位而非均值；任一族低于阈值则整体不通过；
   「行为相似」与「内部相似」两维取 min）。
4. 通过线：综合人类行为相似度 ≥ 80%，且无单族塌方、无表演型得分、无不可重建得分。
5. 反作弊：评估只读、不得为提分篡改 runtime（第 7.7 节）；任一终局能力靠 fallback/降级/prompt 表演冒充即整体判不通过（第 12.7 节）。

实施前置：本规格落为正式 requirement 包前，须再做一次 SOTA 复核（EQ-Bench/EmoBench/MME-Emotion、
LongMemEval-V2/LoCoMo、NeuroAI Turing Test、Rulers/locked-rubric 的最新版本），并产出具体 rubric 与
≥ 300 条（3 族 × 100）真实刺激库。

## 14. 实现方案选型必须服务终局目标

从现在起，每一项实现方案选型（用 LLM、还是小模型、还是确定性方程、还是 NN）都必须显式服务终局目标，并遵守以下选型原则：

1. **内容/判断分离原则**：模型（LLM / 小模型 / NN）只负责提供内容与自评估证据；owner 始终拥有最终判断、状态与契约。自进化不得让模型夺走 owner 的判断权。
2. **必须用大 LLM 的场景**：11 内部思考（已落地）、13 工具选择与规划（function-calling）、14 自我修订起草。这些是开放式生成与规划，确定性方法无法替代。
3. **必须用 embedding / 小模型的场景**：06 记忆表征与 10 定向检索（向量检索）、03 的 novelty（= 与记忆表征的距离）、replay 打分。优先本地小模型以支持自训练与离线再训练。
4. **应当用确定性方程 + 可学习参数、而非黑盒 NN 的场景**：04 神经调质、05 内感受体感、09 门控阈值。这些要求稳定、可解释、不发散，参数通过 `P5` 的 RL 在线微调演化，契约中已预留 `mandatory_learned_parameters`。
5. **自训练分层落地**：L1 记忆式学习（持久化 + RAG，最先做、最安全、不训练任何东西即有成长感）→ L2 可学习参数 RL → L3 小模型离线蒸馏/再训练 → L4 受治理行为自修订 → L5 受治理代码自修改。层级越高风险越大，必须在低层安全门成立后才推进。
6. **奖励信号锚点**：自训练的奖励信号以 `brain.mmd` 已画出的多巴胺奖励预测误差（reward prediction error）为主锚点，由真实运行后果（执行成败、连续性是否被推进、目标冲突是否缓解）定义，而非人工硬编码分数。
7. 任何选型若不能说明它推进了第 12 节哪条终局验收标准，就不应被采纳为终局方向上的实现。

## 15. 终局目标下对实现推进的具体要求

第 9 节的五个问题在终局目标下扩充为七个问题。从现在起，每项实现推进必须能回答：

1. 当前变更属于哪个 requirement owner。
2. 当前新增状态是否已经是正式 contract，而不是临时结构。
3. 当前实现是纯骨架、baseline 还是相对完整，证据是什么。
4. 这项实现属于第 13 节的哪个阶段（P0–P7），它如何推动该阶段的退出信号成立。
5. 这项实现推进了第 12 节哪一条终局验收标准（FG-1 到 FG-6）。
6. 当前变更是否引入了新的架构债或 owner 越界。
7. 若本项涉及自进化能力，它是否落在 14 治理 + 测试 + 评估 + 21 可观测构成的适应度门之内，且可回滚。

如果某项开发不能清楚回答这七个问题，它就不应被视为合格的 Helios 终局推进。

## 16. 最终判定语句

只有当 Helios 已经成为一个以 `brain.mmd` 因果链为真实工作方式、以真实演化的情感为内部底色、以可只读重建的自我意识为中轴、以受治理的工具使用为效应出口、并且能在治理 + 测试 + 评估 + 可观测构成的适应度门内自训练并自我开发的持续存在类脑系统，且第 12 节锁定的 `FG-1` 到 `FG-6` 全部以可证伪方式成立时，才允许宣称它达到了 **Helios 终局目标**。

`Helios v2.0.0`（P1 里程碑）只是这条路上的内部闭环成立门。在终局目标达成之前，任何阶段性成果都只能被视为朝终局推进的实现切片，而不能被误称为最终完成态。第 12 节的终局验收标准一经锁定，不得以放宽定义、prompt 表演或绕过 owner / 测试 / 评估 / 治理的方式规避。
